# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Проект: помощник для собеседований

Цель: считывать звук встречи (системный аудиовыход) и микрофона, транскрибировать его и отвечать на вопросы **в реальном времени**.

## Команды

```powershell
uv sync                                            # установка зависимостей
uv run pytest -q                                   # все тесты
uv run pytest -q tests/test_answer.py -k question  # один тест
uv run ruff format . ; uv run ruff check --fix .   # формат + линт
uv run python -m interview_helper.web              # веб-UI на http://localhost:8765
uv run python -m interview_helper.main --input-file <wav>  # отладочный прогон на файле
uv run python -m interview_helper.main --list-devices      # аудиоустройства (WASAPI)
uv run python -m interview_helper.bench <wav>      # бенчмарк латентности
```

Живой запуск (`main` без `--input-file`) включает захват микрофона — его запускает пользователь, не Claude (страхует хук guard_run).

Код лежит в `src/interview_helper/`: `capture.py` (аудио: callback-режим PyAudio — блокирующий read нельзя, WASAPI loopback при тишине не отдаёт кадров и вешает/роняет процесс; устройства — только WASAPI host API) → `transcribe.py` (whisper) → `answer.py` (детекция вопроса + `claude -p`) → `pipeline.py` (общий цикл с колбэком событий). Транскрипт и ответы каждой сессии пишутся в `sessions/<время>.md` (`session_log.py`) — это вход для отчётов Фазы 1. Дефолт ответов — sonnet + effort low (haiku слабее держит контекст, medium+ слишком многословен для реального времени). Первые реплики после старта транскрибируются медленно — прогрев whisper; это ожидаемо. Поверх пайплайна два фронтенда: `main.py` (терминал, rich) и `web.py` + `static/index.html` (FastAPI + SSE, порт 8765). Оценка ресурсов whisper — в `docs/resources.md`.

## Стек

- **Транскрипция:** faster-whisper (локально, стриминг по чанкам).
- **Ответы:** Claude Haiku через `claude -p` (headless-режим Claude Code CLI). Пользователь использует инструмент сам для себя — это не нарушает ToS.
- **Платформа:** Windows 11. Захват системного звука — через WASAPI loopback (например, `soundcard` или `pyaudiowpatch`), микрофон — обычный устройство ввода.

## Дорожная карта

- **MVP:** рабочий инструмент без предварительного контекста — захват звука, транскрипция, ответы в реальном времени.
- **Фаза 1 — сделано:** `context/vacancy.md` + `context/resume.md` подмешиваются в промпт суфлёра (`answer.load_context`); пост-сессионный отчёт (`report.py`, кнопка «Отчёт» в UI и скилл `/session-report`) генерирует саммари/Q&A/оценку в `sessions/<время>.report.md`.
- **Фаза 2:** контекст по пользователю, накопление знаний от сессии к сессии (профиль, слабые места, история ответов).

## Архитектура (план)

Конвейер в реальном времени:

1. **Захват аудио** — два параллельных потока: loopback (звук встречи) и микрофон. Важно помечать источник — вопросы интервьюера приходят из loopback.
2. **Транскрипция** — faster-whisper обрабатывает чанки с VAD; латентность важнее полноты, поэтому маленькая модель (`small`/`base`) с `compute_type="int8"` предпочтительнее точной, но медленной. Против «мусора на экране» (`transcribe.py`): `condition_on_previous_text=False`, чтобы whisper не зацикливал галлюцинацию; сегменты с высоким `no_speech_prob` и низким `avg_logprob` отсекаются; короткие «титры»-галлюцинации («Продолжение следует…», «Спасибо за просмотр!» и т.п.) выкидываются по блоклисту. Дефолт языка сессии в веб-UI — русский (автоопределение на шумном аудио подмешивает посторонние языки).
3. **Детекция вопроса + ответ** — в фоновом потоке отправляются в `claude -p --model haiku` с накопленным контекстом разговора. Не блокирует транскрипцию — новый вопрос отмечает старый ответ. Промпт лаконичен (SYSTEM_PROMPT + транскрипт до 6000 символов), используется `--output-format stream-json` и `--include-partial-messages` для латентности. Дешёвая эвристика `is_question` может ложно срабатывать на репликах без запроса информации; поэтому параллельно основному ответу (не последовательно — латентность на грани) запускается триаж-Haiku (`Answerer.triage`): вердикт `NO` убивает свой процесс ответа и шлёт `answer_discard`, стирая баббл в UI. Триаж fail-open (ошибка/пустой ответ → отвечаем) и не применяется к ручному запросу из UI («→ LLM»).
4. **UI** — вывод ответа пользователю по мере генерации (терминал или оверлей). Каждый ответ отслеживается по ID, так что перекрывающиеся вопросы отображаются корректно.

Ключевые ограничения, влияющие на все решения: сквозная латентность (звук → ответ на экране) и устойчивость к перекрывающейся речи из двух источников.
